Fix: Race Time Estimate UI — Align with Backend Predictor Logic Problem The race predictor backend was fixed to exclude walk/run intervals and use VO2-max-led estimates, but the training plan gauge in RaceTimeEstimate.tsx still uses the old local calculation that includes contaminated data. Result: backend says 29:30, UI shows 47:27. Root cause: RaceTimeEstimate.tsx calculates locally from recent completed/planned workouts without applying the same contamination filters as the backend. Solution Rewrite the local estimate logic in RaceTimeEstimate.tsx to mirror the backend race-predict filters and weights, so the UI matches the fixed predictor. Changes to src/components/RaceTimeEstimate.tsx 1. Add contamination filter (same as backend) Create helper isCleanContinuousRun that rejects activities matching ANY: - Title/name regex: /walk|w\/r|w\+r|run\/walk|interval|fartlek|rep(s|eats)?/i - Average pace >8:30/km - Lap variance CV >0.30 (if lap data available) - Note: HR sample check skipped in frontend (no access to raw HR samples in typical UI state) Apply this filter to any activities/workouts used for the gauge calculation. 2. Prioritize VO2 max over pace data Pull latest VO2 max from daily_metrics (last 14 days). If VO2 max exists: - Use vo2to5k lookup table (same as backend) to get baseline 5K time - Apply Riegel formula to scale to target distance - This becomes the PRIMARY estimate (60% weight minimum) If clean continuous runs exist: - Calculate tempo/easy pace from filtered activities only - These become SECONDARY support (30% tempo, 10% easy max) If no clean runs AND no VO2 max: - Show "Insufficient data for estimate" - Do NOT show a prediction based on walk/run sessions 3. Sanity cap (same as backend) After computing estimate T: - Derive T_vo2 from VO2 max + Riegel - If T > T_vo2  *1.40:*  Replace T with T_vo2  *Show warning: "Estimate based on VO2 max — recent training includes walk/run sessions" 4. Walk/run phase handling If all recent activities are filtered out (walk/run phase): - Use VO2 max at 100% weight - Show note: "Estimate based on VO2 max — tempo data will improve as continuous running develops" - Do NOT average walk/run session paces 5. UI display changes Replace the current estimate display with:* *Primary number:** The VO2-max-led / clean-run estimate (28:30-30:30 range) **Basis note (small text under estimate):** - "Based on VO2 max 42" (if VO2-only) - "Based on VO2 max 42 + recent tempo runs" (if clean tempo data exists) - "Based on clean continuous runs" (if no VO2 max but clean runs exist) **Warning banner (yellow, if contaminated data was filtered):** "⚠️ Recent walk/run sessions excluded from estimate — prediction will improve as continuous running develops" **Breakdown disclosure (expandable/tooltip):** Show which data sources contributed: - "VO2 max 42: predicts 29:30 (60% weight)" - "Tempo pace: excluded (walk/run sessions)" - "Easy pace: excluded (contaminated by walk breaks)" 6. Remove stale dependencies Remove any code that: - Averages "completed planned sessions" without filtering - Uses planned workout target paces as primary estimate - Treats walk/run interval pace as valid tempo data The gauge should ONLY use: - Latest VO2 max (primary) - Clean continuous runs (secondary support) - Backend race predictor results (if available/cached) 7. Fetch VO2 max from daily_metrics Add query at component mount: ```typescript const { data: vo2Data } = await supabase .from('daily_metrics') .select('vo2_max') .eq('user_id', [user.id](http://user.id)) .not('vo2_max', 'is', null) .order('date', { ascending: false }) .limit(1) .single(); ``` Use this as the primary input to the estimate calculation. 8. Alignment check After fix, verify: - Backend predictor (via chatbot): "What's my 5K time?" → 28:30-30:30 - Training plan gauge (UI): → 28:30-30:30 - Both should match within ±2 minutes for same VO2 max and activity data Expected result for VO2 max 42 in walk/run phase **Before fix:** - UI shows: 47:27 (contaminated by walk/run averages) - Backend shows: 29:30 (correctly uses VO2 max only) - Mismatch destroys trust **After fix:** - UI shows: 28:30-30:30 (VO2-max-led estimate) - Backend shows: 28:30-30:30 (same logic) - Note: "Based on VO2 max 42 — tempo data will improve as continuous running develops" - Warning: "⚠️ Recent walk/run sessions excluded from estimate" - Match builds confidence Technical notes - Reuse vo2to5k lookup table (should already be available as constant/util) - Riegel exponent 1.06 (same as backend) - If VO2 max not available, fall back to clean-runs-only estimate (don't block UI) - No backend changes needed (this is pure frontend alignment) - No new API calls except fetching VO2 max from daily_metrics Out of scope - No changes to backend race-predict (already fixed) - No changes to training plan generation - No changes to workout sync or [intervals.icu](http://intervals.icu) integration - No database migrations Files to change - src/components/RaceTimeEstimate.tsx (primary changes) - src/lib/race-calculations.ts or similar (if creating shared vo2to5k helper) - src/pages/TrainingPlan.tsx (only if VO2 max needs to be passed as prop) Verification 1.
